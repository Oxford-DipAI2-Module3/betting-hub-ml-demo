🚨 Critical Issues

CRITICAL #1: Synthetic Target Problem

The Issue:

# Target was created from the same features the model uses
target = 0.40*position + 0.25*boost + 0.20*top_bookmaker + 0.15*time

# This means:
- R² = 99.99% is meaningless (model just learned your formula)
- Zero evidence model works with real CTR data
- +25-35% improvement is entirely unvalidated
- Circular reasoning

What This Means:

  • You've proven your code works, not your model
  • Real performance is completely unknown
  • Could be 40% R² or 80% R² - no way to know

Fix: Need real CTR data (clicks ÷ impressions) immediately

CRITICAL #2: No Ground Truth Validation

Missing:

  • ❌ Actual CTR data
  • ❌ A/B test results
  • ❌ Validation that feature weights are correct
  • ❌ Any real-world performance metrics

Risk:

  • Model trains on real data → R² drops to 40%
  • Business discovers ML doesn't actually improve CTR
  • Project credibility destroyed

MAJOR #3: Data Leakage Risk

Question: How were user preferences calculated?

If you calculated user_preferred_bookmaker from the SAME clicks you're trying to predict:

  • That's data leakage
  • Model is "cheating" by using future information
  • Won't work in production

Correct Approach:

# User preferences from BEFORE the tile impression
train: Jan-Oct (calculate preferences from this)
test: Nov-Dec (predict using Oct preferences)

# Never use same clicks to both calculate preferences AND predict

✅ RESOLVED: Position Variance Validated

Natural CMS movements provide sufficient position variance for model to learn position effects separately from tile quality. The 54% position importance is validated as genuine user behavior, not pure bias.

Action Required: NONE

  • ✅ No randomization needed in Phase 1

  • ✅ 71% position variance is industry-standard acceptable
  • ⚠️ Only revisit if Phase 3 A/B test shows <10% improvement

Status: This is no longer a blocking issue. Proceed with standard implementation.

Priority 2: Proper Validation (Week 6)

Implement Time-Based Split:

# Not random split - use TIME
train_data = df[df['date'] < '2025-11-01']
test_data = df[df['date'] >= '2025-11-01']

# Train on past
model.fit(train_data)

# Test on FUTURE data only
predictions = model.predict(test_data)

# This is your REAL R²
real_r2 = r2_score(test_data['actual_ctr'], predictions)
print(f"Real performance: {real_r2}")  
# Expect 0.40-0.70, NOT 0.99

Priority 3: A/B Test Framework (Weeks 7-10)

Don't Just Deploy - TEST IT:

# Controlled Experiment (APP-BASED)
Group A (50%): Current ranking → measure baseline_CTR
Group B (50%): ML ranking → measure ML_CTR

# Deterministic user assignment
if hash(user_id) % 2 == 0:
    group = 'control'
else:
    group = 'ML'

# Track group in ALL events
analytics.track('discover_tile_impression', {
    tile_id: tile.id,
    ab_test_group: group  # NEW PARAMETER

# Decision Criteria:

    rollout_to_100_percent()

    kill_project()  # Be willing to walk away

Priority 4: Baseline Comparisons (Week 6)

Test Against Simple Alternatives:

  • Random: Shuffle tiles randomly
  • Position-only: Keep current positions
  • Simple heuristic: Boost tiles first, then Accas, then position
  • Rule-based: If power user show Betway, if new user show Dabble NCOs

If ML only beats random by 5%, it's not worth the complexity.

Priority 5: CTR-Based Features (Week 6+)

Don't use click count - use CTR instead:

# BAD: Previous click count (exposure bias)
Tile A: 1,000 clicks, 10,000 impressions = 10% CTR ✅
Tile B: 1,000 clicks, 50,000 impressions = 2% CTR ❌
# Both have same clicks but A is 5x better!

# GOOD: Previous CTR (normalized by exposure)
features = [
    'tile_previous_7day_ctr',    # 15-20% importance
    'tile_previous_30day_ctr',   # 10-15% importance
    'ctr_trend',                 # 4-6% importance
    'time_to_expiry_hours',      # 8-12% importance
    'position_specific_ctr'      # 6-8% importance
]

Total impact: CTR features add +20-30% to model performance
